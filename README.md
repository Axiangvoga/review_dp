# 本地生活平台

基于 SpringBoot + Redis + MySQL 构建的高性能后端系统。
针对本地生活场景中常见的 “数据库压力激增”、“高并发超卖” 以及 “接口恶意刷单” 等痛点，本项目通过 Redis 滑动窗口限流、Lua 脚本原子扣减及多级缓存策略重构了核心链路。

以下为hmdp学习中的思考与优化🤔：

### 指数退避重试

在调用高德地图等第三方 API 时，网络波动或瞬时限流是无法回避的常态。若在并发环境下简单的“原地重试”，往往会引发“**惊群效应**”——即大量失败请求在同一时间点再次集中冲击，导致本就拥堵的网络彻底瘫痪。

为此，引入了指数退避重试机制作为兜底：

当前请求失败是不立即重试而是让重试等待时间随重试次数呈指数递增，并且添加上随机波动避免请求二次碰撞。


### 登录窗口限流

在本地生活场景中，登录接口是流量的第一道入口，也是最容易遭受 “暴力破解” 和 “瞬时高并发” 攻击的重灾区。限流可以防止暴力破解、保护DB、保证资源公平访问。主流限流算法横向对比：

1.固定窗口：存在临界突刺问题，直接pass

2.令牌桶算法：恒定速率产生令牌，取到令牌才处理，允许一定量突发流量

3.漏桶算法：强行平滑流量，无法保证正常用户体验，pass

4.滑动窗口：窗口随时间移动，细分统计区间，解决了临界突刺问题，平滑度与精度的平衡点，暂选方案

令牌桶允许积攒令牌应对突发流量，这可能给攻击者留下瞬间爆破的窗口，而滑动窗口通过 Redis 的 ZSet 结构对时间区间进行细分审计，能严格确保任意时刻往前推的周期内请求数均在安全阈值内，既能以极简的代码实现分布式全局限流，又在防范恶意攻击与保护 DB 稳定性之间取得了最佳平衡。

### 缓存策略

1.热点 Key 失效（缓存击穿）
 
对于平台中商铺的信息其实是一个变动比较少的、读多写少的数据，因此比较适合写入 Redis 缓存中，降低数据库的压力。

当一个热点 Key（比如秒杀活动详情）TTL 过期的瞬间，几千个请求同时涌入。如果直接查数据库重建缓存，数据库瞬间就会崩掉。这就是缓存击穿。我调研了两种主流方案：

互斥锁方案：当缓存失效时，让所有请求去抢锁，抢到锁的那一个线程去查数据库重建缓存，其他线程休眠等待。
逻辑过期方案：不给 Redis Key 设置物理 TTL，而是把过期时间存在 Value 里。线程取出数据后判断是否过期。如果过期了，获取互斥锁，开启一个独立线程去后台更新缓存，自己直接返回旧数据。
方案对比（CAP 定理的抉择）：

互斥锁：保证了一致性，牺牲了可用性。
逻辑过期：保证了可用性，而牺牲了一致性。
针对这个（活动信息）或（热搜榜单），它其实是一种社交媒体类型的数据，对于一致性的要求并不是强一致性，用户看到的热搜榜单晚了 2 秒更新，其实无伤大雅（AP 模型，重可用性）。

最终结论：我为了保证服务的可用性，选择了使用逻辑过期的方案来解决缓存击穿问题。

2.查询不存在的数据（缓存穿透）
对于平台中商铺的信息其实是一个变动比较少的、读多写少的数据，因此比较适合写入 Redis 缓存中，降低数据库的压力，如果有人恶意用脚本疯狂请求不存在的商铺 ID，请求就会直接穿透 Redis 打到数据库。

因此需要解决缓存穿透的问题，我了解到两种解决方案：一是使用缓存空值、二是使用布隆过滤器。

方案对比：

缓存空值：实现简单，虽然存在额外内存占用，但可以通过设置较短时间的 TTL 解决。
布隆过滤器：存在误判可能性，且实现复杂，需要引入新的组件，需要维护。
针对这个商铺信息查询，因为目前的平台的商铺数量并不是很大。为了解决缓存穿透问题，我选择了缓存空值的方案来解决缓存穿透问题。
